# AI 知识库 V1 差距追踪

## 文档目的

这份文档用于持续记录项目当前能力、尚未完善的模块、以及后续开发优先级。
后续每一轮有实质性开发或优化时，都应同步更新本文件，避免项目状态只存在于聊天记录里。

当前评估时间：

- 2026-04-18

---

## 一、已形成的主能力

### 1. 聊天主链路

- 已支持会话列表、会话详情、会话删除
- 已支持会话记忆：`sessionSummary`、`sessionFacts`
- 已支持消息 `thinking`、`sources`、`toolCalls`
- 已支持会话导出 Markdown / PDF
- 已支持页面切换后聊天继续在前端后台执行

### 2. 检索与问答

- 已支持知识库检索
- 已支持 query rewrite
- 已支持 summary chunk 优先、raw chunk 兜底
- 已支持 retrieval debug 接口与前端调试入口

### 3. 文档工作台

- 已支持文档上传
- 已支持解析文本预览与编辑
- 已支持总结资产独立查看与编辑
- 已支持 AI 自动总结、启发式摘要、人工编辑摘要
- 已支持失败重试、状态刷新、分类切换

### 4. 文档任务机制

- 已支持异步文档处理任务
- 已支持处理状态流转：
  - `UPLOADED`
  - `PARSING`
  - `SUMMARIZING`
  - `INDEXING`
  - `COMPLETED`
  - `FAILED`
  - `CANCELED`
- 已支持任务列表、任务取消、任务重试

### 5. 知识库与知识图谱

- 已支持知识库 CRUD
- 已支持分类 CRUD 与分类树
- 已支持知识图谱接口
- 已能基于分类、标签、文档引用和 `[[wikilink]]` 构建知识网络

### 6. 设置与模型配置

- 已支持用户头像、AI 助手头像、bio
- 已支持 LLM provider 管理与默认配置
- 已接入本地 embedding 模型

## 二、当前未完善模块总览

### 1. 聊天与工具层

当前状态：

- 已有 `ChatToolService` 基础版
- 已支持在聊天结果中展示工具执行卡片
- 本轮已补齐“无 LLM 配置时可直接返回工具结果”的降级链路
- 本轮已扩展更多高频动作：
  - 重命名知识库
  - 删除知识库
  - 查看知识库统计
  - 检查文档是否已存在
  - 重命名分类
  - 删除分类

仍未完善的点：

- 工具意图识别仍以关键词和规则匹配为主，复杂表达容易误判
- 工具动作覆盖面还不够，缺少文档创建/移动/批量操作等更深层管理动作
- 工具结果虽然已展示，但距离“可审计、可恢复、可扩展”的产品化状态还有差距
- 还没有完整的 tool registry、schema 管理和权限控制体系

判断：

- 这是当前最值得继续开发的模块之一，而且本轮已经完成一轮可用性增强

### 2. 文档任务治理

当前状态：

- 已有异步任务模型和前端任务中心
- 已支持取消、重试、状态轮询

仍未完善的点：

- 任务中心还偏基础列表视图，缺少更强的筛选、分组、历史归档和审计信息
- 后端虽然已有取消机制，但“真正稳定的任务级中断”还需要继续验证和打磨
- 失败信息还不够结构化，诊断体验可以更好

判断：

- 属于“已可用，但还不是成熟治理模块”

### 3. 文档工作台

当前状态：

- 解析文本、总结资产、总结方式切换都已成型
- 文档工作台已经具备可用性

仍未完善的点：

- PDF / Office 文档的高保真预览和结构化读取还不够强
- 长文档总结在质量、稳定性、超时处理上还有提升空间
- 文档阅读体验和原始文件视图还没有完全统一

判断：

- 已经是可用模块，但还未达到高质量内容工作台标准

### 4. 检索链路与解释性

当前状态：

- 已有 retrieval debug
- 已能展示 query rewrite 和命中结果

仍未完善的点：

- 检索调试更偏向研发面板，还不是完全产品化的解释性功能
- 缺少更系统的召回对比、命中分析、失败原因解释
- 查询改写、召回结果、最终答案三者之间的关系还可以展示得更清楚

判断：

- 研发可用性不错，但用户可理解性还不够强

### 5. 知识图谱 / 双向链接网络

当前状态：

- 后端图谱生成能力已经存在
- 说明“Obsidian 风格知识网络”已有底层基础

仍未完善的点：

- 前端图谱交互体验还需继续确认和增强
- 关系类型相对基础，更多语义关系还没有建立
- 从图到文档、从文档到图的双向导航体验还可以更完整

判断：

- 是“有基础，但还没成为完整知识导航系统”的模块

### 6. 设置与个人资料

当前状态：

- 头像、AI 助手头像、bio、LLM 配置都已可用

仍未完善的点：

- 缺少更多偏好设置，如主题、语言、AI 风格、默认行为
- 账户中心的产品化程度还不够高

判断：

- 属于“基础可用，但还不是完整用户中心”

### 7. 界面与文案一致性

当前状态：

- 主要工作台页面已经具备统一产品方向

仍未完善的点：

- 仓库内仍有明显的中文编码历史债务
- 多个页面、文档、部分文案存在乱码或风格不一致的情况
- 全局视觉语言还可以再做一轮统一清理

判断：

- 不是主功能缺陷，但属于明显的产品化阻力

### 8. 架构与后续版本能力

当前状态：

- 当前单体结构内的 service boundary 已比较清楚

仍未完善的点：

- 更强的 PDF / Office 结构化解析
- 多模态支持
- 完整的 function calling / tool registry 体系
- 缓存、归档、清理和更大规模的服务化演进

判断：

- 这些属于下一阶段能力建设，不是当前 V1 的直接阻塞项

## 四、建议的下一步开发顺序

### Step 1

继续补聊天工具层，重点往下做：

- 更多知识库管理动作
- 更稳的工具意图识别
- 更清晰的工具失败反馈
- 逐步抽离成更正式的 tool registry / tool execution 结构

### Step 2

继续完善任务治理，包括更稳定的任务取消和更好的失败诊断。

### Step 3

做一轮编码和文案清理，降低后续维护成本。

### Step 4

增强知识图谱与检索解释层，让“知识库可导航、可理解”更完整。

## 六、文档维护规则

- 哪些能力已完成
- 哪些能力从“未完成”变成“部分完成”
- 当前剩余缺口
- 下一轮优先级是否发生变化

如果一轮开发改变了行为、接口、路由、存储模型或交付边界，也应同步更新对应文档：

- `docs/requirements.md`
- `docs/architecture.md`

## 2026-04-18 增量记录：知识库编辑链路补齐

### 新完成
- 后端补充 `PUT /knowledge-base/{id}`，支持按用户权限更新知识库名称和描述。
- 前端 `KnowledgeBase.vue` 已从只读占位提示改为真实编辑弹窗。
- Pinia store 与 `frontend/src/api/knowledge.ts` 已补齐知识库更新调用。

### 仍待继续
- 聊天工具层仍缺少更完整的文档创建、移动、批量操作能力。
- 历史中文编码问题仍影响部分页面与文档，可作为独立治理项继续清理。

## 2026-04-18 增量记录：聊天工具层文档移动与批量操作

### 新完成
- `ChatToolService` 新增文档移动能力，可把一篇或多篇文档移动到指定分类，也支持移动到“未分类”。
- `ChatToolService` 新增文档批量删除能力，能按用户消息中的多个文档名批量执行删除。
- `DocumentService` 补充了按标题匹配文档、批量改分类、批量删除的复用能力，后续前端批量操作也可直接复用。

### 仍待继续
- 批量操作目前仍以“按标题匹配”为主，后续还可以增加更稳的结构化 schema、显式确认和更强的歧义处理。
- 聊天工具层还缺少文档批量重命名、跨知识库移动、批量标签管理等更深层动作。

## 2026-04-18 增量记录：Retrieval Debug 修复与增强

### 新完成
- 修复了 Retrieval Debug 读取 Redis 向量数据时 `String` 被错误当作 `byte[]` 处理的问题，兼容旧数据中 Base64 字符串形式的 embedding。
- 重写 `RetrievalDebug.vue` 页面，清理历史乱码文案，补齐错误态、空态、命中统计和更清晰的对比布局。
- 页面不再额外弹重复错误消息，改为以内联错误提示展示调试失败原因。

### 仍待继续
- Retrieval Debug 目前仍是单问题调试页，后续可以补召回阈值、排序策略、summary/raw 占比等更多可调参数。
- 还缺少对“为什么最终选择 rewritten query”的更显式解释，以及对命中差异的高亮对比。

## 2026-04-18 增量记录：Knowledge Base 页面模板修复

### 仍待继续

## 2026-04-18 增量记录：导航与知识库页乱码清理

### 新完成
- 清理了 `KnowledgeBase.vue` 中仍然可见的按钮文案乱码。
- 清理并重写了 `Layout.vue` 中侧边栏说明、快捷入口说明和顶部 route 描述中的历史乱码文本。

## 2026-04-18 增量记录：聊天工具层确认与歧义保护

### 新完成
- `ChatToolService` 为删除知识库、删除分类、批量删除文档增加了显式确认门槛；用户没有明确表达“确认删除 / 确定删除”时，工具将拒绝执行。
- `ChatToolService` 为知识库重命名、知识库删除、分类重命名、分类删除、文档移动、文档批量删除补上了歧义检测；短名称匹配到多个候选时，会先返回候选列表而不是直接命中第一个。
- `DocumentService` 新增按知识库枚举全部文档的能力，供聊天工具层做更稳妥的目标解析与批量操作判断。

### 仍待继续
- 当前确认机制仍然是规则驱动，后续可以进一步抽象成统一的 tool policy / permission 机制。
- 当前歧义解析仍以完整名和包含匹配为主，后续可以补充别名、拼音、标签和上下文联合解析。

## 2026-04-18 增量记录：工具状态与权限语义继续结构化

### 新完成
- `ChatToolService` 新增轻量级 `ToolPolicy` 注册信息，开始把“是否破坏性操作、是否需要确认”从分支判断里抽离出来。
- 删除类工具在聊天结果中不再与普通失败混在一起，而是返回 `blocked` 状态，明确表示“系统正在等待用户确认”。
- `Chat.vue` 已适配 `blocked` 状态，前端会以“待确认”标签和警示色展示工具卡片。

### 仍待继续
- 当前 `ToolPolicy` 还是轻量内嵌实现，后续仍可继续演进为更正式的 tool registry / schema / permission 模块。

## 2026-04-18 增量记录：工具执行分发继续结构化

### 新完成
- `ChatToolService.execute()` 已拆分为知识库、文档、分类三个处理段，后续继续接 tool registry 时不再需要在单个超长方法里维护全部逻辑。
- 前端 `chat.ts` 已把工具状态收敛为 `success / failed / blocked` 联合类型，聊天页的工具状态展示改成了显式类型驱动。
- 当前聊天页已通过 `vue-tsc -b` 校验，工具卡片状态扩展没有引入新的前端类型错误。

### 仍待继续
- 后端当前仍缺少独立的 Java 编译校验环境；后续应补上 Maven wrapper 或统一本地构建入口，降低迭代验证成本。

## 2026-04-18 增量记录：工具定义开始收口

### 新完成
- `ChatToolService` 新增统一的 `ToolDefinition` 映射，把工具名称、展示标题、权限策略收口到同一处维护。
- 知识库、文档、分类相关工具调用已开始改用统一工具常量与定义解析，减少标题文案和工具名在分支中的重复散落。
- 现有 `addSuccessToolCall / addFailedToolCall / addBlockedToolCall` 已支持通过工具定义自动补全标题，为后续继续抽 registry 做好兼容层。

### 仍待继续
- 当前参数解析与意图判断仍然散落在 `ChatToolService` 内部，下一步可以继续抽成独立 parser / resolver。
